# Гайд для новичка: как понять и защитить AituDesk

> Этот документ объясняет проект простым языком — с аналогиями из повседневной жизни. Прочитав его, вы сможете уверенно рассказать о проекте на защите.

---

## Что это такое — простыми словами

**AituDesk** — это «электронная приёмная» IT-отдела колледжа.

Представьте: у студента сломался компьютер, не работает Wi-Fi, нужна программа. Раньше он шёл в кабинет айтишника или писал в WhatsApp. Проблемы очевидны:
- Заявки теряются
- Непонятно, кто и когда решит
- Нет истории обращений
- Руководство не видит статистику

**AituDesk решает это так:**
1. Пользователь заходит на сайт и пишет заявку — как «письмо в поддержку»
2. Система **сама** назначает свободного специалиста — как умная очередь в банке
3. Специалист и пользователь общаются в **реал-тайм чате** — как в мессенджере, но привязанном к заявке
4. Всё фиксируется: время создания, время решения, оценка — как чек из магазина
5. Руководство видит **дашборд** с графиками и может скачать **PDF-отчёт**

## Как устроена архитектура

### Трёхзвенная (three-tier) архитектура

```
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│   Браузер   │  HTTP   │   Сервер    │  SQL    │    База     │
│  (React)    │ ◄─────► │  (Express)  │ ◄─────► │ (PostgreSQL)│
│  порт 7754  │  + WS   │  порт 4000  │         │  порт 5432  │
└─────────────┘         └─────────────┘         └─────────────┘
    Frontend               Backend                 Database
  «что видишь»           «что думает»           «что помнит»
```

Почему три слоя, а не один?
- **Разделение ответственности** — каждый делает своё дело. Фронтенд не знает, как хранятся данные. База не знает, как выглядит кнопка.
- **Масштабируемость** — можно запустить 5 копий бэкенда, а база останется одна.
- **Безопасность** — пользователь никогда не общается с базой напрямую.

### Что делает каждый слой

| Слой | Что делает | Файлы |
|---|---|---|
| **Frontend** | Рисует интерфейс, отправляет запросы, показывает данные | `frontend/src/` — React-компоненты, страницы, стили |
| **Backend** | Проверяет пароли, считает SLA, назначает агентов, генерирует PDF, шлёт уведомления | `backend/src/` — роуты, middleware, сервисы |
| **Database** | Хранит данные в таблицах: users, tickets, messages, ratings, notifications | `backend/prisma/schema.prisma` — схема всех таблиц |

## Роли — зачем три типа пользователей

Аналогия — **больница**:

| Роль | Аналогия | Что может |
|---|---|---|
| **USER** | Пациент | Создаёт заявку, описывает проблему, видит только свои, переписывается с агентом, ставит оценку |
| **AGENT** | Врач | Принимает заявки, меняет статусы, пишет публичные ответы и внутренние заметки (невидимые для пользователя) |
| **ADMIN** | Главврач | Всё что выше + переназначает заявки, управляет ролями, видит полную аналитику, редактирует базу знаний |

Зачем разделение? Чтобы обычный пользователь не мог случайно закрыть чужую заявку или увидеть внутренние обсуждения IT-отдела.

## Жизненный цикл тикета

Аналогия — **заказ в ресторане**:

```
NEW ──► IN_PROGRESS ──► WAITING ──► RESOLVED ──► CLOSED
 ▲           │             │            │
 └───────────┴─────────────┴── REOPENED ┘
```

| Статус | Аналогия | Что происходит |
|---|---|---|
| `NEW` | Заказ принят | Заявка создана, ищем свободного повара (агента) |
| `IN_PROGRESS` | Готовят | Агент взял в работу, SLA-таймер тикает |
| `WAITING` | Ждём уточнение | Агент спросил — ждёт ответа от пользователя. Таймер на паузе |
| `RESOLVED` | Блюдо готово | Агент считает задачу решённой. Пользователь может оценить |
| `CLOSED` | Счёт оплачен | Пользователь подтвердил. Тикет в архиве |
| `REOPENED` | Вернули блюдо | Пользователь не доволен — заявка снова в работе |

## SLA — что это и зачем

**SLA** (Service Level Agreement) — «обещание скорости». Как доставка пиццы: если заказ срочный — привезут за 30 минут, если обычный — за час.

| Приоритет | Аналогия | Реакция | Решение |
|---|---|---|---|
| `CRITICAL` | Пожар в серверной | 30 мин | 1 час |
| `HIGH` | Не работает почта у отдела | 1 час | 4 часа |
| `MEDIUM` | Не печатает принтер | 4 часа | 24 часа |
| `LOW` | Нужен новый кабель | 24 часа | 72 часа |

Фоновый процесс каждую минуту проверяет: «не просрочен ли дедлайн?». Если да — красный флаг `slaBreached = true` + уведомление агенту и администратору. Это как будильник для IT-отдела.

## Реал-тайм: как работают сокеты

**Обычный HTTP** — это почта: отправил письмо → ждёшь ответ → связь разорвана.

**Socket.IO** — это телефонный звонок: линия открыта постоянно. Результат:
- Агент пишет ответ → пользователь видит **мгновенно** (не надо обновлять страницу)
- Кто-то печатает → видно «Иванов печатает…»
- Статус тикета сменился → обновляется везде в реальном времени
- Новое уведомление → колокольчик звенит

## Безопасность — 6 уровней защиты

| № | Механизм | Аналогия |
|---|---|---|
| 1 | **bcrypt** — хеширование паролей | Пароль превращается в «кашу из символов». Даже если украдут базу — пароли не прочитать |
| 2 | **JWT Access Token** (15 мин) | Дневной пропуск — быстро истекает, если украдут — не страшно |
| 3 | **Refresh Token** (7 дней, httpOnly cookie) | Карта для получения нового пропуска. JavaScript не может её прочитать — только браузер пересылает серверу |
| 4 | **Zod-валидация** | Охранник на входе — проверяет каждое поле: email правильный? Пароль >= 6 символов? Оценка от 1 до 5? |
| 5 | **Helmet** | Каска — добавляет защитные HTTP-заголовки (XSS, clickjacking, sniffing) |
| 6 | **RBAC** | Табличка «Только для персонала» — USER не может зайти на страницу ADMIN, даже зная URL |

## Мониторинг — зачем Prometheus и Grafana

Аналогия — **приборная панель автомобиля**. Под капотом не видно, но датчики показывают:

| Метрика | Аналогия |
|---|---|
| `http_requests_total` | Тахометр — сколько «оборотов» делает сервер |
| `http_request_duration_seconds` | Спидометр — как быстро отвечает |
| `tickets_total` / `tickets_resolved_total` | Одометр — сколько «проехали» |
| `aitudesk_nodejs_heap_size_total_bytes` | Датчик температуры — не перегревается ли |

**Prometheus** = бортовой компьютер (собирает цифры каждые 15 секунд).
**Grafana** = экран на приборной панели (рисует графики из этих цифр).

## Юнит-тесты — зачем и как

Аналогия — **экзамен с автопроверкой**. Преподаватель (Vitest) даёт 80+ задач, студент (сервер) отвечает, и за 5 секунд видно: всё правильно или вот тут ошибка.

**Зачем?** Вы написали новую функцию — откуда знать, что не сломали старую? Тесты проверяют это автоматически.

**Как работает без базы?** Вместо настоящей PostgreSQL используется «муляж» (mock). Мы говорим: «когда спросят пользователя — верни вот этого тестового». Как манекен на витрине: выглядит настоящим, но не требует зарплаты.

```
Реальность:  Браузер → Express → Prisma → PostgreSQL → ответ
Тест:        Supertest → Express → Mock (данные в памяти) → ответ
```

## Docker — зачем 6 контейнеров

**Без Docker**: установи Node.js нужной версии, PostgreSQL, настрой переменные, пути, пароли. На другом компьютере — всё заново. «У меня работает, а у тебя нет».

**С Docker**: `docker compose up -d --build` — и через 2 минуты работает **на любом компьютере**. Каждый сервис в своей изолированной «комнате»:

| Контейнер | Аналогия | Что делает |
|---|---|---|
| `postgres` | Архив | Хранит все данные |
| `backend` | Офис | API, логика, WebSocket |
| `frontend` | Ресепшн | Nginx раздаёт сайт, проксирует запросы к бэкенду |
| `pgadmin` | Библиотекарь | Веб-интерфейс для просмотра БД |
| `prometheus` | Бортовой компьютер | Собирает метрики |
| `grafana` | Приборная панель | Рисует графики |

## Ключевые паттерны в коде

### 1. Компонентный подход (React)
Интерфейс разбит на маленькие переиспользуемые блоки: `<StatCard>`, `<StatusBadge>`, `<PriorityBadge>`. Как LEGO — из одних и тех же кубиков собираем разные страницы.

### 2. REST API
Бэкенд предоставляет набор «адресов» (эндпоинтов):
- `GET /api/tickets` — получить список тикетов
- `POST /api/tickets` — создать новый
- `PUT /api/tickets/:id/status` — обновить статус

Каждый адрес делает ровно одно действие. Как меню в ресторане — каждое блюдо под своим номером.

### 3. Middleware (промежуточный слой)
Запрос проходит через цепочку проверок, прежде чем попасть к обработчику:
```
Запрос → [Helmet] → [CORS] → [Auth JWT] → [Zod Validation] → Обработчик → Ответ
```
Как проходная на заводе: сначала пропуск, потом досмотр, потом — к станку.

### 4. Zustand (управление состоянием)
Фронтенд хранит данные (кто залогинен, тема, уведомления) в «магазинах» (stores). Любой компонент может прочитать или обновить данные. Как общая доска объявлений — все видят актуальную информацию.

## Частые вопросы на защите

**В: Почему React, а не Vue/Angular?**
О: React — самая популярная UI-библиотека (по npm-загрузкам). Компонентный подход, огромная экосистема, легко масштабируется. Для Service Desk нужно много переиспользуемых компонентов (карточки, бейджи, формы) — React идеален.

**В: Зачем TypeScript?**
О: Ловит ошибки до запуска. В проекте ~100 файлов — без типизации легко передать строку вместо числа и узнать об этом только когда сервер упадёт в продакшене.

**В: Зачем Prisma, если можно SQL?**
О: ORM-слой даёт автодополнение в IDE, типизацию, автомиграции и защиту от SQL-инъекций. Вместо `SELECT * FROM users WHERE id = '...'` пишем `prisma.user.findUnique({ where: { id } })`.

**В: Как обеспечена безопасность?**
О: 6 уровней: bcrypt-хеши, JWT 15 мин, refresh в httpOnly cookie, Zod-валидация, Helmet-заголовки, RBAC на каждом маршруте.

**В: Что будет если сервер упадёт?**
О: Docker `restart: unless-stopped` — авторестарт. Prometheus зафиксирует даунтайм. Данные на volume — переживут перезапуск контейнера.

**В: Как масштабировать?**
О: Горизонтально: несколько бэкендов за Nginx load balancer. Socket.IO с Redis-адаптером. PostgreSQL — read replicas.

**В: Зачем юнит-тесты?**
О: 80+ проверок за 5 секунд. Если новая функция сломает старую — увидим сразу, а не через неделю от пользователей.

**В: Как работает авто-назначение?**
О: Система смотрит специализации агентов (HARDWARE, SOFTWARE, NETWORK), находит подходящих, считает у кого меньше активных тикетов — и назначает ему. Балансировка нагрузки.

**В: Почему Docker Compose, а не Kubernetes?**
О: Для учебного проекта / малого деплоя Compose достаточно. Kubernetes нужен при десятках серверов и сотнях контейнеров — это следующий шаг при росте.

**В: Чем отличается от обычного чата (Telegram)?**
О: Структура. Каждая заявка имеет статус, приоритет, дедлайн (SLA), назначенного агента, историю, оценку. В Telegram это всё теряется в потоке сообщений.
