## Короткая речь на 1–2 минуты

> Моя дипломная работа называется **AituDesk**. Это Service Desk система для колледжа, которая помогает IT-отделу принимать, распределять и контролировать заявки пользователей.  
> Основная проблема — хаотичная обработка обращений: заявки могут приходить устно или в мессенджерах, теряться, оставаться без ответственного исполнителя и без контроля сроков.  
> В моей системе есть три роли: пользователь, агент и администратор. Пользователь создаёт заявку, система автоматически назначает подходящего агента, агент общается с пользователем в реал-тайм чате и меняет статус заявки.  
> Дополнительно реализованы SLA-контроль, уведомления, база знаний, AI-ассистент на основе RAG, PDF-отчёты, мониторинг через Prometheus и Grafana, а также **104 автоматических теста**.  
> Проект полностью контейнеризирован через Docker Compose и запускается одной командой.

## Что это такое — простыми словами

**AituDesk** — это «электронная приёмная» IT-отдела колледжа.

Представьте: у студента сломался компьютер, не работает Wi-Fi, нужна программа. Раньше он шёл в кабинет айтишника или писал в WhatsApp. Проблемы очевидны:
- Заявки теряются
- Непонятно, кто и когда решит
- Нет истории обращений
- Руководство не видит статистику

**AituDesk решает это так:**
1. Пользователь заходит на сайт и пишет заявку — как «письмо в поддержку»
2. Система **сама** назначает свободного специалиста — как умная очередь в банке
3. Специалист и пользователь общаются в **реал-тайм чате** — как в мессенджере, но привязанном к заявке
4. Всё фиксируется: время создания, время решения, оценка — как чек из магазина
5. Руководство видит **дашборд** с графиками и может скачать **PDF-отчёт**

## Как устроена архитектура

### Трёхзвенная (three-tier) архитектура

```
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│   Браузер   │  HTTP   │   Сервер    │  SQL    │    База     │
│  (React)    │ ◄─────► │  (Express)  │ ◄─────► │ (PostgreSQL)│
│  порт 7754  │  + WS   │  порт 4000  │         │  порт 5432  │
└─────────────┘         └─────────────┘         └─────────────┘
    Frontend               Backend                 Database
  «что видишь»           «что думает»           «что помнит»
```

Почему три слоя, а не один?
- **Разделение ответственности** — каждый делает своё дело. Фронтенд не знает, как хранятся данные. База не знает, как выглядит кнопка.
- **Масштабируемость** — можно запустить 5 копий бэкенда, а база останется одна.
- **Безопасность** — пользователь никогда не общается с базой напрямую.

### Что делает каждый слой

| Слой | Что делает | Файлы |
|---|---|---|
| **Frontend** | Рисует интерфейс, отправляет запросы, показывает данные | `frontend/src/` — React-компоненты, страницы, стили |
| **Backend** | Проверяет пароли, считает SLA, назначает агентов, генерирует PDF, шлёт уведомления | `backend/src/` — роуты, middleware, сервисы |
| **Database** | Хранит данные в таблицах: users, tickets, messages, ratings, notifications | `backend/prisma/schema.prisma` — схема всех таблиц |

## Роли — зачем три типа пользователей

Аналогия — **больница**:

| Роль | Аналогия | Что может |
|---|---|---|
| **USER** | Пациент | Создаёт заявку, описывает проблему, видит только свои, переписывается с агентом, ставит оценку |
| **AGENT** | Врач | Принимает заявки, меняет статусы, пишет публичные ответы и внутренние заметки (невидимые для пользователя) |
| **ADMIN** | Главврач | Всё что выше + переназначает заявки, управляет ролями, видит полную аналитику, редактирует базу знаний |

Зачем разделение? Чтобы обычный пользователь не мог случайно закрыть чужую заявку или увидеть внутренние обсуждения IT-отдела.

## Жизненный цикл тикета

Аналогия — **заказ в ресторане**:

```
NEW ──► IN_PROGRESS ──► WAITING ──► RESOLVED ──► CLOSED
 ▲           │             │            │
 └───────────┴─────────────┴── REOPENED ┘
```

| Статус | Аналогия | Что происходит |
|---|---|---|
| `NEW` | Заказ принят | Заявка создана, ищем свободного повара (агента) |
| `IN_PROGRESS` | Готовят | Агент взял в работу, SLA-таймер тикает |
| `WAITING` | Ждём уточнение | Агент спросил — ждёт ответа от пользователя. Таймер на паузе |
| `RESOLVED` | Блюдо готово | Агент считает задачу решённой. Пользователь может оценить |
| `CLOSED` | Счёт оплачен | Пользователь подтвердил. Тикет в архиве |
| `REOPENED` | Вернули блюдо | Пользователь не доволен — заявка снова в работе |

## SLA — что это и зачем

**SLA** (Service Level Agreement) — «обещание скорости». Как доставка пиццы: чем срочнее заказ, тем быстрее должны отреагировать и решить проблему.

| Приоритет | Аналогия | Реакция | Решение |
|---|---|---|---|
| `CRITICAL` | Пожар в серверной | 2 часа | 4 часа |
| `HIGH` | Не работает почта у отдела | 4 часа | 8 часов |
| `MEDIUM` | Не печатает принтер | 8 часов | 24 часа |
| `LOW` | Нужен новый кабель | 24 часа | 72 часа |

Фоновый процесс каждую минуту проверяет: «не просрочен ли дедлайн?». Если да — красный флаг `slaBreached = true` + уведомление агенту и администратору. Это как будильник для IT-отдела.

## Реал-тайм: как работают сокеты

**Обычный HTTP** — это почта: отправил письмо → ждёшь ответ → связь разорвана.

**Socket.IO** — это телефонный звонок: линия открыта постоянно. Результат:
- Агент пишет ответ → пользователь видит **мгновенно** (не надо обновлять страницу)
- Кто-то печатает → видно «Иванов печатает…»
- Статус тикета сменился → обновляется везде в реальном времени
- Новое уведомление → колокольчик звенит

## Безопасность — 6 уровней защиты

| № | Механизм | Аналогия |
|---|---|---|
| 1 | **bcrypt** — хеширование паролей | Пароль превращается в «кашу из символов». Даже если украдут базу — пароли не прочитать |
| 2 | **JWT Access Token** (15 мин) | Дневной пропуск — быстро истекает, если украдут — не страшно |
| 3 | **Refresh Token** (7 дней, httpOnly cookie) | Карта для получения нового пропуска. JavaScript не может её прочитать — только браузер пересылает серверу |
| 4 | **Zod-валидация** | Охранник на входе — проверяет каждое поле: email правильный? Пароль >= 6 символов? Оценка от 1 до 5? |
| 5 | **Helmet** | Каска — добавляет защитные HTTP-заголовки (XSS, clickjacking, sniffing) |
| 6 | **RBAC** | Табличка «Только для персонала» — USER не может зайти на страницу ADMIN, даже зная URL |

## Мониторинг — зачем Prometheus и Grafana

Аналогия — **приборная панель автомобиля**. Под капотом не видно, но датчики показывают:

| Метрика | Аналогия |
|---|---|
| `http_requests_total` | Тахометр — сколько «оборотов» делает сервер |
| `http_request_duration_seconds` | Спидометр — как быстро отвечает |
| `tickets_total` / `tickets_resolved_total` | Одометр — сколько «проехали» |
| `aitudesk_nodejs_heap_size_total_bytes` | Датчик температуры — не перегревается ли |

**Prometheus** = бортовой компьютер (собирает цифры каждые 15 секунд).
**Grafana** = экран на приборной панели (рисует графики из этих цифр).

## Database Exporter Pattern — как объяснить бизнес-метрики

Обычный вариант — увеличивать счётчики в коде при создании или закрытии тикета. Но у такого подхода есть минусы:

- Если backend перезапустился — значения в памяти могут обнулиться.
- Если запустить несколько backend-инстансов — у каждого будут свои счётчики.
- Если данные изменили напрямую в БД — счётчик в памяти этого не узнает.

В AituDesk для бизнес-метрик используется другой подход:

```text
Prometheus scrape /metrics
→ prom-client вызывает async collect()
→ backend делает SELECT COUNT(*) через Prisma
→ отдаёт актуальное значение из PostgreSQL
```

Сильная фраза для защиты:

> Для бизнес-метрик я использую базу данных как single source of truth. Поэтому количество тикетов по статусам, категориям, приоритетам и SLA считается напрямую из PostgreSQL при каждом scrape Prometheus.

## AI-ассистент — как объяснить RAG

AI-ассистент — это не отдельная магия, а обычный backend endpoint:

```text
Пользователь → POST /api/ai/chat
→ поиск релевантных статей в базе знаний
→ сбор контекста
→ Groq API через OpenAI-compatible SDK
→ ответ + список источников
```

Что важно сказать:

- **Groq** — текущий AI-провайдер.
- **Модель** — `llama-3.3-70b-versatile`.
- **RAG** — модель получает контекст из базы знаний.
- **Источники** — backend возвращает список найденных статей.
- **Retry** — при 429/5xx есть повторные попытки с backoff.
- **Провайдер можно заменить** — достаточно поменять `AI_API_KEY`, `AI_BASE_URL`, `AI_MODEL`.

Сильная фраза:

> AI не заменяет базу знаний. База знаний остаётся источником истины, а AI помогает быстро найти и объяснить нужную информацию пользователю.

## PDF-отчёт — как объяснить Worker Thread

PDF-отчёт содержит:

- созданные тикеты;
- закрытые тикеты;
- открытые тикеты;
- среднее время решения;
- среднюю удовлетворённость;
- нарушения SLA;
- разбивку по категориям;
- разбивку по приоритетам.

Почему используется Worker Thread:

> Node.js выполняет JavaScript в основном потоке event loop. Генерация PDF может быть тяжёлой операцией: шрифты, текст, таблицы, сохранение файла. Поэтому генерация вынесена в отдельный worker thread, чтобы основной сервер продолжал отвечать на HTTP-запросы.

Что сказать про дизайн PDF:

> PDF сделан в том же editorial-стиле, что и интерфейс: русская типографика PT Sans, §-секции, цветные KPI-карточки и горизонтальные бары.

## Юнит-тесты — зачем и как

Аналогия — **экзамен с автопроверкой**. Преподаватель (Vitest) даёт 104 задачи, студент (сервер) отвечает, и за несколько секунд видно: всё правильно или вот тут ошибка.

**Зачем?** Вы написали новую функцию — откуда знать, что не сломали старую? Тесты проверяют это автоматически.

**Как работает без базы?** Вместо настоящей PostgreSQL используется «муляж» (mock). Мы говорим: «когда спросят пользователя — верни вот этого тестового». Как манекен на витрине: выглядит настоящим, но не требует зарплаты.

```
Реальность:  Браузер → Express → Prisma → PostgreSQL → ответ
Тест:        Supertest → Express → Mock (данные в памяти) → ответ
```

## Docker — зачем 6 контейнеров

**Без Docker**: установи Node.js нужной версии, PostgreSQL, настрой переменные, пути, пароли. На другом компьютере — всё заново. «У меня работает, а у тебя нет».

**С Docker**: `docker compose up -d --build` — и через 2 минуты работает **на любом компьютере**. Каждый сервис в своей изолированной «комнате»:

| Контейнер | Аналогия | Что делает |
|---|---|---|
| `postgres` | Архив | Хранит все данные |
| `backend` | Офис | API, логика, WebSocket |
| `frontend` | Ресепшн | Nginx раздаёт сайт, проксирует запросы к бэкенду |
| `pgadmin` | Библиотекарь | Веб-интерфейс для просмотра БД |
| `prometheus` | Бортовой компьютер | Собирает метрики |
| `grafana` | Приборная панель | Рисует графики |

## Ключевые паттерны в коде

### 1. Компонентный подход (React)
Интерфейс разбит на маленькие переиспользуемые блоки: `<StatCard>`, `<StatusBadge>`, `<PriorityBadge>`. Как LEGO — из одних и тех же кубиков собираем разные страницы.

### 2. REST API
Бэкенд предоставляет набор «адресов» (эндпоинтов):
- `GET /api/tickets` — получить список тикетов
- `POST /api/tickets` — создать новый
- `PUT /api/tickets/:id/status` — обновить статус

Каждый адрес делает ровно одно действие. Как меню в ресторане — каждое блюдо под своим номером.

### 3. Middleware (промежуточный слой)
Запрос проходит через цепочку проверок, прежде чем попасть к обработчику:
```
Запрос → [Helmet] → [CORS] → [Auth JWT] → [Zod Validation] → Обработчик → Ответ
```
Как проходная на заводе: сначала пропуск, потом досмотр, потом — к станку.

### 4. Zustand (управление состоянием)
Фронтенд хранит данные (кто залогинен, тема, уведомления) в «магазинах» (stores). Любой компонент может прочитать или обновить данные. Как общая доска объявлений — все видят актуальную информацию.

## Частые вопросы на защите

### Общие вопросы

**В: Что делает ваш проект?**  
О: Это Service Desk система для колледжа. Она позволяет пользователям создавать IT-заявки, агентам обрабатывать их, а администраторам контролировать сроки, аналитику, пользователей, базу знаний и отчёты.

**В: В чём актуальность?**  
О: В колледже обращения в IT-отдел часто идут устно или через мессенджеры. Из-за этого теряется история, непонятно кто отвечает, нет SLA и статистики. AituDesk централизует этот процесс.

**В: Чем отличается от обычного чата или Telegram?**  
О: В Telegram нет статусов, приоритетов, дедлайнов SLA, назначенного исполнителя, ролей, PDF-отчётов и аналитики. AituDesk — это не просто переписка, а управление процессом поддержки.

### Архитектура и стек

**В: Почему React, а не Vue/Angular?**  
О: React хорошо подходит для сложных интерфейсов с большим количеством компонентов: карточки, таблицы, фильтры, формы, dashboard. Компонентный подход позволяет переиспользовать UI и быстрее развивать проект.

**В: Зачем TypeScript?**  
О: TypeScript ловит ошибки до запуска. В проекте много сущностей: user, ticket, message, notification, rating. Типизация помогает не перепутать поля и быстрее находить ошибки.

**В: Зачем Prisma, если можно SQL?**  
О: ORM-слой даёт автодополнение в IDE, типизацию, автомиграции и защиту от SQL-инъекций. Вместо `SELECT * FROM users WHERE id = '...'` пишем `prisma.user.findUnique({ where: { id } })`.

**В: Почему PostgreSQL?**  
О: В проекте много связанных таблиц: пользователи, тикеты, сообщения, вложения, оценки, уведомления, статьи базы знаний. Для таких связей реляционная база подходит хорошо.

**В: Почему Docker Compose, а не Kubernetes?**  
О: Для учебного проекта и локального демо Compose достаточно. Kubernetes нужен при большом количестве серверов и сложной инфраструктуре. При росте это может быть следующим этапом.

### Безопасность

**В: Как обеспечена безопасность?**  
О: 6 уровней: bcrypt-хеши, JWT 15 мин, refresh в httpOnly cookie, Zod-валидация, Helmet-заголовки, RBAC на каждом маршруте.

**В: Почему refresh token в httpOnly cookie?**  
О: httpOnly cookie нельзя прочитать через JavaScript в браузере. Это снижает риск кражи refresh token при XSS-атаке.

**В: Обычный пользователь сможет открыть чужую заявку через URL?**  
О: Нет. Проверка доступа есть на backend. Даже если пользователь вручную отправит запрос, сервер проверит роль и `creatorId`.

### Бизнес-логика

**В: Как работает авто-назначение?**  
О: Система смотрит специализации агентов (HARDWARE, SOFTWARE, NETWORK), находит подходящих, считает у кого меньше активных тикетов — и назначает ему. Балансировка нагрузки.

**В: Почему пользователь не может сам поставить `RESOLVED`?**  
О: `RESOLVED` означает, что агент выполнил работу. Пользователь может подтвердить решение через `CLOSED` или вернуть заявку в работу через `REOPENED`.

**В: Что такое INTERNAL-сообщения?**  
О: Это внутренние заметки для агентов и администратора. Пользователь их не видит. Они нужны для технических комментариев внутри IT-отдела.

**В: Что будет если сервер упадёт?**  
О: Данные лежат в PostgreSQL volume и переживают перезапуск. Docker может перезапустить контейнер, а Prometheus/Grafana покажут проблему по метрикам.

### AI

**В: Что такое RAG?**  
О: RAG — это подход, где AI сначала получает релевантные данные из базы знаний, а потом отвечает на основе этого контекста. Это снижает риск выдуманных ответов.

**В: Почему Groq?**  
О: Groq даёт быстрый OpenAI-compatible API. Код использует OpenAI SDK с настраиваемым `baseURL`, поэтому можно заменить провайдера на OpenRouter, Gemini, NVIDIA или OpenAI без переписывания логики.

**В: Что будет если AI API недоступен?**  
О: Основная Service Desk система продолжит работать. AI — внешняя интеграция, для неё есть обработка ошибок и retry при 429/5xx. Пользователь увидит понятное сообщение.

### Мониторинг

**В: Зачем Prometheus и Grafana в дипломном проекте?**  
О: Это показывает production-подход. Важно не только сделать функционал, но и наблюдать за системой: RPS, latency, память, event loop, SLA и бизнес-метрики.

**В: Что такое Database Exporter Pattern?**  
О: Это когда бизнес-метрики считаются напрямую из PostgreSQL при scrape Prometheus. Такой подход не теряет значения при рестарте backend и корректен при нескольких инстансах.

**В: Зачем нормализовать роуты в метриках?**  
О: Чтобы Prometheus не создавал отдельную серию для каждого ID. `/api/tickets/abc123` превращается в `/api/tickets/:id`, и cardinality остаётся контролируемой.

### PDF и тесты

**В: Зачем Worker Thread для PDF?**  
О: Генерация PDF может блокировать event loop. Worker Thread переносит тяжёлую работу в отдельный поток, а основной сервер продолжает отвечать на запросы.

**В: Зачем юнит-тесты?**  
О: 104 проверки за несколько секунд. Если новая функция сломает старую, мы увидим это сразу, а не после ручного тестирования или жалоб пользователей.

**В: Почему тесты без реальной базы?**  
О: Для API-тестов Prisma заменяется mock-объектом. Это быстрее, стабильнее и не требует PostgreSQL. Логика Express проверяется через Supertest.

### Масштабирование

**В: Как масштабировать?**  
О: Горизонтально: несколько backend-инстансов за load balancer. Для Socket.IO нужно добавить Redis adapter. PostgreSQL можно вынести на отдельный сервер и настроить backups/read replicas.

**В: Что бы вы улучшили дальше?**  
О: Добавил бы Redis для Socket.IO при масштабировании, S3-хранилище для вложений, CI/CD, email-уведомления, HTTPS-домен и хранение секретов через vault/secrets manager.

## Если что-то пошло не так на защите

**AI не отвечает**  
Сказать: AI зависит от внешнего API, интернета и лимитов провайдера. Основная система автономна и продолжает работать. В коде есть retry и понятная обработка ошибок.

**Grafana пустая**  
Сказать: Prometheus собирает данные каждые 15 секунд. Нужно сделать несколько запросов в приложении: открыть страницы, создать заявку, скачать отчёт — после этого графики появятся.

**PDF не скачивается**  
Сказать: endpoint защищён JWT. Нужно войти как авторизованный пользователь и скачать отчёт через dashboard. Генерация идёт через Worker Thread, есть fallback.

**Забыли ответ на вопрос**  
Сказать: «В текущей версии это не реализовано, потому что фокус диплома был на Service Desk workflow: заявки, роли, SLA, real-time, отчёты и мониторинг. Архитектура позволяет добавить это следующим этапом».
